# 程序员打字竞速游戏 MVP Spec

## 产品定位
这是一个面向程序员的桌面打字练习游戏。它不是通用英文打字器，而是专门训练代码输入能力，重点覆盖括号、引号、大小写、缩进、操作符和常见关键字。首版目标是做出一个“愿意每天打开练 1 局”的单机产品，而不是社交或竞技平台。

## 目标用户
主要用户是有英文键盘输入需求的程序员、学生和技术从业者。默认场景是物理键盘、英文输入法、单人本地练习。首版不支持中文录入练习，不考虑输入法组合态判定。

## 核心玩法
玩家从主菜单进入一局练习，选择时长、语言和难度后开始。每局先显示 3 秒倒计时，然后进入持续 5 分钟或 10 分钟的输入阶段。游戏持续推送短代码片段，玩家逐字符输入，系统实时显示正确、错误、当前位置、剩余时间、WPM、准确率、连击和当前得分。时间结束后立即结算，展示本局结果、历史最佳和近几次成绩趋势。

## 玩法规则
题目以短代码片段连续出现，每段 1 到 8 行，不让玩家面对整页长文。输入判定按字符进行。正确字符推进光标，错误字符即时记录并以红色反馈显示。默认允许退格修改，但错误次数仍会被统计，用于准确率和稳定性评分。片段完成后无缝切到下一段，整局计时不断。若窗口失焦，游戏暂停计时并提示点击恢复，避免长局被误伤。

## 模式与配置
首版只提供两个时长：`5 分钟` 和 `10 分钟`。语言先支持 `JavaScript`、`TypeScript`、`Python`。难度至少分为 `Easy / Medium / Hard`。设置项包含声音开关、默认时长、默认语言、默认难度。所有设置和成绩都保存在本地。

## 内容系统
每个题目片段都带元数据。建议结构如下：

`Snippet = { id, language, difficulty, title, content, tags }`

其中 `tags` 可用于标记训练重点，例如 `array`、`async`、`loop`、`string`、`indentation`、`operator`。首版题库建议每个语言每个难度至少 20 到 30 段，总量先做到 180 段左右，足够支撑 10 分钟模式且重复率可控。出题逻辑按当前筛选条件随机抽取，并尽量避免短时间内重复同一片段。

## 计分与指标
核心展示指标有：

- `Raw WPM`：按总输入字符计算的原始速度
- `WPM`：按正确字符计算的有效速度
- `Accuracy`：正确字符占总输入字符的比例
- `Score`：综合速度、准确率和稳定性的总分
- `Max Combo`：最长连续正确字符数
- `Completed Snippets`：完成的片段数量

推荐计分方向：

- 速度是基础分
- 高准确率有乘区加成
- 长时间稳定输入提供额外奖励
- 高频错误和长时间停顿会削弱总分

不建议首版做过度复杂的街机倍率系统，因为时长是 5 到 10 分钟，重点应该是可持续输入体验，而不是短爆发。

## 状态流转
应用状态可以固定为：

- `menu`
- `countdown`
- `playing`
- `paused`
- `result`

局内状态至少要包含：

- `RunConfig`
- `RunStats`
- `currentSnippet`
- `currentIndex`
- `timeRemainingMs`
- `isFocused`
- `queueOfNextSnippets`

结果记录结构建议为：

`ResultRecord = { id, finishedAt, config, stats }`

## UI 结构
界面建议拆成 5 个区域：

- 顶部状态栏：时长、语言、难度、剩余时间
- 中央主练习区：当前代码片段，带逐字符高亮
- 右侧或下方 HUD：WPM、准确率、得分、连击
- 底部提示区：当前训练标签、阶段提示、暂停提示
- 结果页：本局数据、最佳成绩、最近记录

React 负责页面结构、配置面板、结果页和本地记录展示。Canvas 负责主练习区渲染，这样字符高亮、滚动和光标控制会更稳。

## 交互细节
必须支持物理键盘输入。默认监听可见字符、Backspace、Tab、Enter。Tab 在题目中如果出现，建议统一替换为两个或四个空格，以避免浏览器默认焦点行为干扰。输入焦点必须稳定，点击开始后游戏区自动获取焦点；若失焦，要明确显示“已暂停，点击继续”。长局中每分钟给一次轻量进度反馈，例如“第 3 分钟，状态稳定”之类的阶段提示，但不打断输入。

## 存档与本地数据
本地至少保存两类数据：

- `settings`
- `results`

建议内容：

- 当前默认模式
- 最近 20 条成绩
- 每个模式下的最佳成绩
- 各语言下的最佳成绩

不需要数据库，首版直接用 `localStorage` 或 Tauri 提供的本地持久化能力即可。MVP 优先 `localStorage`，实现更快。

## 音效与视觉
音效只做最小集：

- 按键命中
- 错误提示
- 倒计时结束
- 局终结算

视觉方向建议偏“开发者工作台”，不要做花哨二次元或传统教育软件风格。可以采用深色背景、亮色语法高亮、清晰等宽字体、轻量动画。长局更重要的是“耐看”和“低疲劳”。

## 技术实现建议
前端用 `Vite + React + TypeScript`。游戏逻辑建议分成这些模块：

- `engine`：状态机、计时、判定、计分
- `content`：题库、筛选、随机队列
- `storage`：设置和成绩持久化
- `render`：Canvas 文本布局和字符渲染
- `ui`：菜单、HUD、结果页、设置页

Tauri 只作为桌面壳，首版不需要写复杂 Rust 逻辑。只要能本地运行和打包即可。

## 验收标准
MVP 完成时，至少要满足这些条件：

- 可以在 Windows 上以桌面应用形式运行
- 可以选择 5 分钟或 10 分钟模式
- 可以选择 JavaScript、TypeScript、Python
- 游戏会连续推送代码片段，整局不会空题或卡死
- 可以实时显示 WPM、Accuracy、Score、Combo
- 时间结束后能看到结果页
- 历史成绩和设置重启后仍保留
- 失焦暂停和恢复可用
- 10 分钟模式下不会出现明显性能掉帧或状态错误

## 明确不做
首版不做：

- 联网排行榜
- 账号系统
- 多人对战
- 云同步
- 中文输入法模式
- 自定义导入题库
- 成就、皮肤、商城这类外围系统

## 推荐开发顺序
1. 搭前端壳和状态机
2. 做单局流程：菜单、倒计时、5 分钟模式、结果页
3. 做字符判定和 Canvas 渲染
4. 接题库与连续推题
5. 加本地成绩与设置
6. 补 10 分钟模式、暂停恢复、视觉打磨
7. 最后接 Tauri 打包
